================================================================================
REALTIME PITCH — ARCHITECTURE v4
Teacher-Student conjoint session: session model, latency preservation,
teacher features, student features.
================================================================================

Context from prior decisions:
  - Single Docker container (architecture_v3).
  - Student's browser runs MPM via WASM (authoritative pitch).
  - Server runs Melodia as advisory octave oracle.
  - No user auth / no accounts yet.

--------------------------------------------------------------------------------
(a) SESSION MODEL — HOW TEACHER AND STUDENT CONNECT
--------------------------------------------------------------------------------

1. CORE PRINCIPLE: THE STUDENT'S PITCH PIPELINE MUST NOT CHANGE

   The student's latency-critical path (mic → WASM MPM → local NoteFeed →
   canvas) is a CLOSED LOOP inside their browser.  Introducing a teacher
   must not inject anything into that loop — no extra WS hops, no waiting
   for the server to relay, no synchronisation barrier.

   The teacher is a READ-ONLY OBSERVER of pitch data that ALREADY EXISTS
   on the server.  The data gets to the server anyway (the advisory
   Melodia channel).  The teacher just subscribes to the same stream.

2. SESSION LIFECYCLE (no accounts, shareable link)

   ┌────────────────────────────────────────────────────────────────┐
   │                                                                │
   │  Student opens /                                              │
   │  → frontend generates a session_id (nanoid, 10 chars)         │
   │  → displays shareable URL:  https://domain/s/{session_id}     │
   │  → opens WS to wss://domain/realtime/stream                   │
   │     first TEXT msg includes: {"session_id": "abc123def0",      │
   │                               "role": "student", ...}         │
   │  → server creates SessionRoom("abc123def0"), attaches the      │
   │    student's WS + RealtimePitchState.                          │
   │                                                                │
   │  Teacher opens /s/{session_id}                                │
   │  → frontend reads session_id from URL                          │
   │  → opens WS to wss://domain/realtime/stream                   │
   │     first TEXT msg: {"session_id": "abc123def0",               │
   │                      "role": "teacher"}                        │
   │  → server finds SessionRoom, attaches teacher's WS as observer.│
   │  → server begins forwarding pitch events to teacher.           │
   │                                                                │
   │  Either party disconnects → room stays alive for 5 min         │
   │    (grace period for refresh / reconnect).                     │
   │  Both gone for 5 min → room is reaped.                         │
   │                                                                │
   └────────────────────────────────────────────────────────────────┘

   Why nanoid instead of UUID: shorter URL, easier to read aloud in a
   lesson ("go to slash s slash a-b-c-1-2-3..."), no dashes.  10 chars
   of base-58 = 4.5 × 10^17 combinations — collision-free at any scale
   you'll operate at.

   Why no auth: you're building a teaching tool, not a social network.
   The session_id IS the credential.  Anyone with the link can observe.
   This is equivalent to a Google Meet link — good enough until you need
   persistent history or student accounts.  If you later add auth, it
   layers ON TOP of this scheme (session_id becomes a token-protected
   resource).

3. SERVER-SIDE SESSION OBJECT

   @dataclass
   class SessionRoom:
       session_id: str
       created_at: float
       student_ws: Optional[WebSocket] = None
       teacher_ws: Optional[WebSocket] = None
       pitch_state: RealtimePitchState = field(default_factory=...)
       student_note_log: deque[PitchEvent] = field(default_factory=...)
         # last 300 s of pitch events, ~14 000 entries
         # serves two purposes:
         #   1. teacher catches up on join (student started singing first)
         #   2. teacher can scroll back in their viewport

   rooms: dict[str, SessionRoom] = {}   # in-memory, single-process.

   When the student's advisory hop arrives and Melodia runs, the result
   is simultaneously:
     a) sent back to student (existing advisory reply)
     b) appended to student_note_log
     c) forwarded to teacher_ws (if connected)

   But the TEACHER gets RICHER data than the student:

4. DATA FLOW — TWO TIERS

   TIER 1: Student → Server (advisory channel, every Kth hop)
     Student sends: PCM binary + {"seq", "client_f0", "client_conf",
                                   "client_midi", "client_note"}
     Server runs Melodia, produces advisory reply.
     Server ALSO forwards the student's CLIENT-SIDE pitch to the teacher:

   TIER 2: Server → Teacher (observer channel, every hop the student reported)
     {
       "type": "student_pitch",
       "seq": N,
       "t_student": 12.384,        // student's capture-time
       "midi": 57.12,              // student's WASM MPM result
       "note": "A3",
       "conf": 0.84,
       "voiced": true,
       "melodia_midi": 57.08,      // server's Melodia (may be null if
       "melodia_agrees": true,     //   this wasn't an advisory hop)
       "octave_corrected": false
     }

   KEY: the teacher sees the student's OWN pitch data, not a re-estimated
   version.  The student's WASM MPM runs every hop; the student sends a
   SUMMARY of every hop (just the {midi, conf, voiced} tuple, ~40 bytes)
   alongside the PCM every Kth hop — or even separately as a lightweight
   JSON stream.  This costs ~1.9 KB/s at 47 hops/s, negligible bandwidth.

   This means: the teacher's viewport is 15–50 ms behind the student's
   (one WAN RTT), but it shows THE SAME DATA the student sees.  No
   discrepancy, no "why does teacher see a different note" confusion.

5. WHY NOT WebRTC / peer-to-peer FOR THE SESSION?

   Tempting — "just DataChannel the pitch events directly." But:
   - STUN/TURN infrastructure for NAT traversal (or use a hosted service).
   - ICE negotiation adds 2–5 s to session start.
   - You need the server ANYWAY for Melodia advisory.
   - Pitch event bandwidth is ~2 KB/s.  This is literally 100× below
     the threshold where P2P saves you anything.

   Your server is already in the loop.  Use it as the relay.  Done.

6. CONCURRENT SESSIONS

   Each SessionRoom owns one RealtimePitchState.  At ~2.4 % CPU/user
   (Melodia advisory) + negligible forwarding overhead, your 4-core VPS
   handles ~100 concurrent sessions.  If you need more, add uvicorn
   workers (each gets its own rooms dict — session_id must route to the
   same worker, use consistent hashing or sticky sessions at the LB).

--------------------------------------------------------------------------------
(b) LATENCY — WHAT CHANGES AND WHAT DOESN'T
--------------------------------------------------------------------------------

1. STUDENT LATENCY: UNCHANGED

   The student's path is still:
     Mic → AudioWorklet → WASM MPM → NoteFeed → Canvas (local, ~25–40 ms)
   Nothing about the teacher session touches this loop.

   The advisory Melodia channel is also unchanged.  The only new work on
   the student's browser is: serialize {midi, conf, voiced} as JSON every
   hop and ws.send() it.  This is ~0.1 ms of main-thread work per hop,
   buried in the existing WS send call.

2. TEACHER LATENCY: ONE WAN RTT BEHIND STUDENT

   Student-to-server RTT/2 + server processing (~0.1 ms relay) +
   server-to-teacher RTT/2.

   If student and teacher are on the same ISP/region:
     Student → server: ~20 ms,  server → teacher: ~20 ms → ~40 ms behind.
   Cross-continent:
     ~60–80 ms behind.

   This is INHERENT to any relay architecture (and to WebRTC DataChannel
   through TURN, so P2P wouldn't help).  It is acceptable for a voice
   lesson because:
     - The teacher watches a scrolling record, not a reaction-time game.
     - 40–80 ms is below the threshold of rhythmic perception (~100 ms).
     - The teacher's viewport shows t_student on the X-axis (capture time),
       so the visual position of each note is CORRECT even though it
       arrives late.  The "now" line on the teacher's viewport is simply
       ~50 ms to the LEFT of the student's "now" line.

3. WHAT WOULD ACTUALLY HURT LATENCY (avoid these)

   BAD IDEA                          WHY
   ─────────────────────────────────  ──────────────────────────────────
   Teacher's browser re-runs WASM    Double compute, no benefit — same
   MPM on the student's PCM.         data, different browser clock. Drift.

   Server re-estimates pitch from    You already have the student's WASM
   student PCM and sends THAT to     result.  Re-estimating adds ~1.5 ms
   the teacher.                      server-side + risks a different
                                     value (teacher sees ≠ student sees).

   Teacher sends "commands" that     Any control message on the pitch
   block student pitch pipeline.     path stalls the student's canvas.
                                     Teacher commands go on a SEPARATE
                                     logical channel (see §c.4).

   Bundling pitch events into        Teacher gets a burst of 10 events
   batches (e.g., 10 per message).   every 200 ms instead of 1 event
                                     every 21 ms.  Scrolling stutters.
                                     Send one JSON per hop.

4. CLOCK SYNCHRONISATION BETWEEN VIEWPORTS

   Problem: student's performance.now() ≠ teacher's performance.now().
   The teacher cannot use t_recv for the X-axis; it must use t_student
   (the capture time the student computed locally).

   Solution: the teacher's viewport maintains a CLOCK OFFSET:
     offset = t_local_recv - t_student_capture - estimated_one_way_delay
   Estimated one-way delay is measured via periodic ping/pong on the WS
   (server timestamps the relay; both sides see server clock).

   This is NOT precision NTP — ±10 ms is fine.  The cumulative record
   scrolls at ~80 px/s; 10 ms = 0.8 px.  Imperceptible.

   On teacher join, the server sends the student_note_log backfill
   (§a.3) as a batch, with all t_student timestamps.  The teacher's
   viewport draws the last N seconds of history, then transitions to
   live streaming.  No gap.

--------------------------------------------------------------------------------
(c) TEACHER FEATURES
--------------------------------------------------------------------------------

The teacher's page loads the SAME app as the student, but with role=teacher.
The viewport is the same scrolling piano-roll canvas — but with overlays
and controls the student doesn't see.

c.1  OBSERVE THE STUDENT'S PITCH — PRIMARY

   The scrolling record shows the student's pitch stream in real time.
   Identical rendering to the student's own viewport (midi → y, t → x,
   confidence → line width, colour by note).

   PLUS: a faint "Melodia advisory" ghost line when available, showing
   the octave oracle's opinion.  This lets the teacher see when the
   system was uncertain about the octave.

c.2  REFERENCE PITCH / TARGET LINE

   The teacher defines a "target" melody that overlays on the student's
   viewport.  Two input modes:

   a) SING IT: teacher's own mic → WASM MPM → a second pitch trace
      drawn as a thin coloured line on the SAME record.  Both traces
      scroll together.  The student sees the teacher's trace overlaid on
      their own.

   b) DRAW IT: teacher clicks/drags on the piano-roll canvas to sketch
      a target contour (a sequence of {t_start, t_end, midi} rects).
      This is sent to the server, relayed to the student, and drawn as
      translucent note blocks behind the student's live trace.

   The student's goal: make their trace match the target.  Visual gap
   between the two traces = the error, immediately obvious.

   Implementation: teacher sends a JSON "overlay" message on the control
   channel.  The server stores it in the SessionRoom and relays it to
   the student.  Tiny bandwidth (a 4-bar melody is ~50 note events,
   <2 KB).

c.3  ANNOTATION / MARKERS

   Teacher can drop timestamped markers on the record:
     - "Good" (green flag) — positive reinforcement at a specific moment.
     - "Check this" (yellow flag) — something to revisit.
     - "Flat" / "Sharp" (arrows) — micro intonation feedback.
     - Free-text (short, <80 chars).

   Markers are relayed to the student in real time and drawn as icons on
   the student's viewport at the corresponding X position.  They persist
   in student_note_log for review.

   Implementation: {"type": "marker", "t_student": ..., "kind": "flat",
   "text": "..."}  — server relays, student draws.  No pitch-path impact.

c.4  CONTROL CHANNEL (separate from pitch stream)

   Teacher actions (overlay, markers, config changes) travel as TEXT
   messages on the teacher's existing WS, tagged {"type": "teacher_cmd",
   "cmd": "..."}.  The server validates, stores in SessionRoom, and
   forwards to the student's WS as {"type": "teacher_event", ...}.

   CRITICAL CONSTRAINT: teacher commands are NEVER injected into the
   student's pitch processing path (process_chunk).  They go directly
   to the student's frontend rendering layer.  The student's pitch
   pipeline doesn't even know a teacher exists.

c.5  STUDENT MIC MONITOR (optional, bandwidth-gated)

   Teacher may want to HEAR the student, not just see the pitch trace.
   Two options:

   OPTION A — Reuse existing PCM stream.
     The student already sends PCM every Kth hop for Melodia.  The server
     can relay that same PCM to the teacher's browser, which plays it via
     AudioContext.  Quality: 48 kHz mono int16, but only every 4th hop
     → ~12 kHz effective with gaps.  Choppy.  Acceptable for "am I
     hearing vibrato or noise?" but not for full monitoring.

   OPTION B — Separate full-rate PCM stream.
     Student sends every hop's PCM (not just every Kth) on a second WS
     or as a binary sub-channel.  ~96 KB/s.  Server relays to teacher.
     Teacher plays back via AudioWorklet.  Latency: 1 WAN RTT + playback
     buffer (~40–80 ms).  Quality: lossless 48 kHz/16-bit mono.

   OPTION C — WebRTC audio for monitoring only.
     If you want low-latency audio monitoring with Opus compression and
     built-in jitter buffering, add a WebRTC peer connection JUST for the
     audio-to-teacher path (SFU not needed — it's 1-to-1).  This exists
     alongside the WS pitch path, not replacing it.  Add this LATER,
     only if teachers ask for it.

   Recommendation: start with OPTION A (it's free — the data already
   flows).  Add OPTION B if teachers complain about choppiness.  Skip C
   until you have real demand.

c.6  SESSION RECORDING / REVIEW

   student_note_log (deque of pitch events, last 300 s) is already on the
   server.  Add a "save session" teacher action that snapshots the log to
   a JSON file (or SQLite row).  The teacher can later load it in a
   review page — same piano-roll viewport, but with seek/scrub controls
   instead of real-time scrolling.

   No audio recording.  Just pitch events + markers + overlays.
   Tiny files (~500 KB for a 30-minute lesson).  No privacy/storage
   burden from raw audio.

--------------------------------------------------------------------------------
(d) STUDENT FEATURES
--------------------------------------------------------------------------------

d.1  LIVE PITCH VIEWPORT — PRIMARY (already designed in v1)

   Scrolling piano-roll cumulative record.  Y = MIDI, X = time,
   colour = note hue, width = confidence.  "Now" line at ~85% from
   the left edge.

   Added for session mode: teacher's overlays (§c.2) and markers (§c.3)
   are drawn on the same canvas, behind the student's live trace.

d.2  CENTS METER (HUD)

   Large display of current note name + cents deviation (-50 to +50).
   This is the most immediately actionable feedback for a singing
   student: "you're on A3, 12 cents flat."

   Implementation: already trivial from the WASM MPM output.
     cents = (midi - Math.round(midi)) * 100

d.3  TARGET MATCH INDICATOR

   When a teacher overlay (§c.2) is active, show a LIVE DELTA between
   the student's current midi and the target midi:
     delta_st = student_midi - target_midi_at_current_time
   Rendered as:
     - Vertical bar (like a VU meter) showing +/- semitones from target.
     - Colour gradient: green (within ±0.25 st) → yellow (±0.5 st) →
       red (>±1 st).
   This gives the student IMMEDIATE corrective feedback without having
   to visually compare two lines on the scrolling chart.

d.4  SELF-ASSESSMENT MODE (no teacher present)

   When the student opens / without a teacher:
     - No session_id in URL → solo mode.
     - No teacher overlays, no markers, no teacher_cmd channel.
     - Student still has the full pitch viewport + cents meter.
     - Optionally: built-in reference melodies (scales, arpeggios,
       simple songs) loaded from a static JSON file.  Student picks one
       from a menu → it draws as the target contour on the piano-roll.
       This works without a teacher — useful for practice between lessons.

d.5  VISUAL FEEDBACK FOR VOICING / SILENCE

   When the student stops singing (voiced=false), the viewport shows a
   subtle grey region instead of a blank gap.  This distinguishes:
     - "I stopped singing" (grey) vs.
     - "the detector lost my pitch" (gap with a preceding confidence drop)
   Helps the student understand whether THEY stopped or the system
   struggled.

d.6  TEACHER PRESENCE INDICATOR

   Small status badge: "Teacher connected" (green dot) / "Teacher
   watching" (when teacher has the tab focused, optional via a focus
   heartbeat) / "Solo mode" (no session).

   Subtle.  No distraction.  The student should be focused on their
   voice, not on whether the teacher is watching.  But knowing someone
   IS watching has a documented motivational effect in pedagogy (the
   Hawthorne effect, productively applied).

d.7  LATENCY INDICATOR

   Small text in HUD corner: "Lag: 23 ms".  This is the student's own
   round-trip to the advisory server, NOT the teacher latency.  It lets
   the student know the system is working.  If it spikes above 100 ms,
   turn it yellow so they know to check their connection.

   Do NOT show the teacher's lag to the student.  It's irrelevant to
   their experience and would just cause anxiety.
